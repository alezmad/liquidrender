## 1. Philosophy

### What Knosia Is

Knosia is not a dashboard. Not a chatbot. Not a BI tool.

**Knosia is a thinking partner** â€” an AI data scientist that:
- Understands your business vocabulary
- Thinks while you sleep
- Collaborates with your team
- Remembers everything

### Design Principles

| Principle | Meaning |
|-----------|---------|
| **Adaptive** | UI responds to user intent, not the other way around |
| **Conversational** | Data exploration through dialogue, not clicks |
| **Collaborative** | Insights are shared, built upon, never siloed |
| **Persistent** | Every conversation becomes institutional knowledge |
| **Proactive** | Knosia surfaces insights before you ask |

### The Vocabulary Problem

Every company has data. Every company has BI tools. Nobody has solved the vocabulary problem.

"Active Users" means different things to:
- Engineering (anyone who logged in)
- Product (completed a core action)
- Sales (paying customer who logged in)
- CEO (whatever the board deck says)

**Knosia's job:** Become the shared vocabulary. Speak everyone's language.

### What Gets Versioned

| Block Type | Version Trigger | Paired Content |
|------------|-----------------|----------------|
| Question | User edits text | Answer auto-versions when Q changes |
| Answer | Regenerate / Q changes | Paired with Question version |
| Note | User edits text | Standalone |
| Chart | Regenerate | Paired with data query |

### User Actions & Version Results

| User Action | Result | Implementation |
|-------------|--------|----------------|
| Edit question | New version of same block | Push to `versions[]`, increment `currentVersionIndex` |
| Click "Branch from here" | New branch + new block | Create `Branch`, create `Block` with `parentBlockId` |
| Click "Branch from v1" | New branch from old version | Create `Branch` with `forkedFromVersionNumber: 1` |
| View old version | Show historical content | Read `versions[n]` |
| Restore old version | Make old version current | Set `currentVersionIndex = n` |

### Versioning Principles

1. **Versions don't create graph complexity** â€” they're internal to a block
2. **Branches are intentional** â€” require explicit user action
3. **Linear stays linear** â€” editing doesn't accidentally branch
4. **History is preserved** â€” but tucked away in dropdown
5. **Time travel is possible** â€” branch from any past version

## 18. Accessibility

### Standards

- **WCAG 2.1 AA** compliance
- **Screen reader** support (ARIA labels)
- **Keyboard navigation** for all actions
- **Focus indicators** visible
- **Color contrast** 4.5:1 minimum

### Keyboard Navigation

| Key | Action |
|-----|--------|
| `Tab` | Move between elements |
| `Enter` | Activate button/link |
| `Esc` | Close modal/cancel |
| `Arrow keys` | Navigate within components |
| `âŒ˜K` | Command palette |

### Screen Reader Considerations

- Charts have text descriptions
- Dynamic content announces changes
- Form errors are announced
- Navigation landmarks defined
