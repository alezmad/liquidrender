# ISS-103: Unicode Operator Rendering in ASCII Prompts

**Category:** Minor Edge Cases
**Priority:** Low
**Status:** Resolved

## Issue

When LiquidCode containing Unicode operators (Δ, §, →, ↑) is displayed in ASCII-only environments (terminal prompts, legacy systems, certain LLM contexts), how should operators be rendered?

## Resolution

### Canonical Rendering Strategy

The specification already defines ASCII equivalents in **B.1.1** (Appendix B: Hardening Specification). Apply these mappings when rendering to ASCII-only contexts:

| Unicode | ASCII Fallback | Context |
|---------|----------------|---------|
| `Δ` | `delta:` | Mutation operations |
| `§` | `signal:` | Signal declarations |
| `→` | `->` | Replacement/flow |
| `↑` | `move:` | Move operations |

### Implementation Requirements

1. **Detection**: Renderers MUST detect ASCII-only output contexts via:
   - Environment flag (`LIQUID_ASCII_ONLY=true`)
   - Terminal capability detection
   - Explicit API parameter (`renderOptions.asciiOnly`)

2. **Normalization**: Use existing `normalizeToASCII()` function (B.1.2) for conversion

3. **Consistency**: When displaying in mixed contexts (e.g., web UI showing terminal command), use ASCII form for the terminal portion

### Example

```liquidcode
// Unicode form (default)
§dateRange:dr=30d,url
Δ+K$profit@[1,2]
Δ↑@[0,0]→[1,1]

// ASCII-only rendering (when context requires)
signal:dateRange:dr=30d,url
delta:+K$profit@[1,2]
delta:move:@[0,0]->[1,1]
```

## Specification Impact

**No specification change required.** The solution is already specified in B.1.1-B.1.2. This resolution simply clarifies rendering behavior for display contexts.
